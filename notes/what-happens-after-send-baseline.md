# What Happens After a User Presses Send in a Local AI Application?

Status: Week 0 baseline, written without external research on 2026-07-18.

Authorship note: Codex prepared this baseline at the learner's request. The learner should annotate and rewrite it in their own words before treating it as a personal skills baseline.

## My current mental model

When a user presses **Send**, the application turns an interaction in the user interface into a request that can be processed by several layers of software. In a local AI application, most or all of those layers run on the user's own computer or on a nearby machine controlled by the same organization. "Local" does not necessarily mean that everything happens inside one process. The browser, application server, model runtime, retrieval service, and data store may be separate processes or containers even when they share one physical computer.

The first event occurs in the client interface. A button handler or keyboard handler reads the text in the prompt field and combines it with other client state, such as the conversation identifier, selected model, attached files, or generation settings. The client normally validates simple conditions before sending anything: the prompt is not empty, the attachment is an allowed type, and the user has not exceeded an obvious size limit. The client then serializes the request, commonly as JSON for text fields and multipart form data when files are included.

If the interface is a browser, the browser constructs an HTTP request to an application endpoint. The destination may be `localhost`, a private network address, or a local domain name. The operating system resolves the destination, chooses a network route, and opens or reuses a socket. With HTTPS, a TLS session protects the request in transit and lets the client check the server's identity. With plain HTTP on localhost, the bytes may not leave the machine, but other local processes and browser security rules still matter.

The operating system moves the request through its networking stack to the listening application process. A web server or framework such as FastAPI accepts the connection, parses the HTTP method, path, headers, and body, and converts the serialized fields into application objects. The server should reject malformed or oversized input before expensive AI work begins. It also identifies the caller through a session, token, certificate, or another mechanism and checks whether that identity is allowed to use the requested model, conversation, documents, and tools.

The application then assembles the work for the model. It loads prior conversation turns, system instructions, user preferences, and any application policy that belongs in the prompt. If the application uses retrieval, it transforms the user's question into a search query or embedding, searches an index, filters results by the user's document permissions, and selects relevant passages. Those passages are added as context. This is a security boundary: retrieving a passage is not enough; the application must prove the user is authorized to see it. Retrieved text is untrusted data and should not gain control merely because it contains instructions.

Before inference, the application converts the assembled text into tokens using the tokenizer associated with the selected model. Tokens are integer identifiers, not words in the ordinary sense. The request includes those token identifiers and generation settings such as maximum output length, temperature, or stop conditions. The model runtime loads or reuses model weights from storage, places required tensors in RAM and possibly VRAM, and performs the mathematical operations of inference.

The model processes the input tokens and produces a distribution of possible next tokens. A decoding procedure chooses one token, appends it to the sequence, and repeats the process until a stop token, length limit, or application rule ends generation. On a GPU, much of the matrix computation runs in parallel. The CPU still coordinates input, memory, scheduling, networking, and parts of the runtime. If memory is insufficient, the runtime may fail, offload work, use slower system memory, or require a smaller or more heavily quantized model.

Some applications allow the model to request tools. The model does not directly change a database or operating system merely by emitting text. The application parses the proposed tool name and arguments, validates them against a schema, checks the user's authorization and any approval requirement, then invokes an ordinary software function. Read-only lookups and write actions should be distinguished. A consequential write should fail closed when approval, identity, or input validation is missing. The tool result can be returned to the model for another generation step, but it remains data rather than trusted instruction.

The model's generated tokens are decoded back into text. The application may stream partial text to the client as it is generated. It can also attach citations, tool results, warnings, or a confidence-related status. A responsible system checks whether cited passages actually support the answer and whether the response violates content, privacy, or authorization rules. The language model itself should not be the only enforcement point because the same component producing the answer cannot be assumed to reliably police every failure mode.

The server packages the result into an HTTP response or a stream of events. The operating system carries the bytes back to the client, and the interface updates the conversation view. The browser renders the text and any structured elements such as citations. The user sees the apparent end of the request, although background work may continue.

Throughout the request, the system should produce telemetry: timings, resource use, errors, model and data versions, retrieval decisions, authorization outcomes, and tool approvals. Logs should make a failure diagnosable without copying sensitive prompts or protected documents unnecessarily. The application may persist the conversation, but retention should be deliberate and governed rather than automatic.

My main uncertainty at this baseline is the exact boundary between responsibilities in a particular implementation. A desktop application may combine the client and server. A model runtime may handle tokenization or expose an OpenAI-compatible API. Retrieval may be embedded inside the application or delegated to another service. The stable idea is that pressing **Send** starts a chain across interface, operating system, network or inter-process communication, application policy, data access, model computation, optional tools, response delivery, and audit. Each boundary can fail independently, and diagnosing the system requires identifying which boundary produced the observed behavior.
